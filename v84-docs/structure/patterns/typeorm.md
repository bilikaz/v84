# TypeORM Patterns

## Entity — strict mode

All entity fields use `!` (definite assignment assertion) because TypeORM populates them, not the constructor:

```typescript
@Entity()
export class Message {
  @PrimaryGeneratedColumn()
  id!: number;

  @Column({ type: 'varchar', length: 36, unique: true })
  key!: string;

  @Column({ type: 'text' })
  body!: string;

  @CreateDateColumn()
  createdAt!: Date;
}
```

## Migrations — ALWAYS generated, NEVER hand-written

Entities are the source of truth. TypeORM diffs entities against the DB and generates migrations automatically. Agents must NEVER write migration SQL by hand — it wastes tokens and introduces drift.

TypeORM CLI needs commonjs mode even when the app uses `nodenext`. Use `typeorm-ts-node-commonjs` wrapper:

```json
{
  "scripts": {
    "migration:generate": "typeorm-ts-node-commonjs -d database/data-source.ts migration:generate",
    "migration:run": "typeorm-ts-node-commonjs -d database/data-source.ts migration:run",
    "migration:revert": "typeorm-ts-node-commonjs -d database/data-source.ts migration:revert",
    "migration:show": "typeorm-ts-node-commonjs -d database/data-source.ts migration:show",
    "seed": "ts-node --compiler-options '{\"module\":\"commonjs\"}' database/seed-runner.ts"
  }
}
```

Do NOT use `pnpm typeorm` directly — it fails with `nodenext` modules. Always use the `typeorm-ts-node-commonjs` wrapper.

Migration generation MUST run inside the Docker container (needs DB access via service name). Use `pnpm exec`:

```bash
# Generate migration inside container
docker compose exec -w /app/apps/api api pnpm exec typeorm-ts-node-commonjs migration:generate src/database/migrations/NameOfChange -d src/database/data-source.ts

# Run migrations inside container
docker compose exec -w /app/apps/api api pnpm exec typeorm-ts-node-commonjs migration:run -d src/database/data-source.ts
```

Do NOT generate migrations from the host — the DB is only accessible inside Docker via the `db` service name.

## DatabaseModule — separate module, not inline

TypeORM config lives in its own module using `forRootAsync` with ConfigService. Uses `__dirname` for entity/migration paths so they resolve correctly at runtime.

```typescript
// src/database/database.module.ts
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { ConfigModule, ConfigService } from '@nestjs/config';

@Module({
  imports: [
    TypeOrmModule.forRootAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (configService: ConfigService) => ({
        type: 'mysql',
        host: configService.get<string>('database.host'),
        port: configService.get<number>('database.port'),
        username: configService.get<string>('database.username'),
        password: configService.get<string>('database.password'),
        database: configService.get<string>('database.name'),
        entities: [__dirname + '/../**/entities/*.entity{.ts,.js}'],
        migrations: [__dirname + '/migrations/*{.ts,.js}'],
        synchronize: false,
        migrationsRun: true,
        timezone: 'Z',
      }),
    }),
  ],
})
export class DatabaseModule {}
```

Key settings:
- `synchronize: false` — NEVER auto-sync schema, use migrations
- `migrationsRun: true` — run pending migrations on app startup
- `__dirname` paths — resolve correctly in both ts-node and compiled dist

## Data source — CLI entry point

Separate data source for TypeORM CLI (migrations, seeds). Lives at `src/database/data-source.ts`. Uses env vars directly (not ConfigService — CLI runs outside NestJS).

```typescript
// src/database/data-source.ts
import 'dotenv/config';
import { DataSource } from 'typeorm';

export const AppDataSource = new DataSource({
  type: 'mysql',
  host: process.env.DATABASE_HOST ?? 'localhost',
  port: parseInt(process.env.DATABASE_PORT ?? '3306', 10),
  username: process.env.DATABASE_USER ?? 'root',
  password: process.env.DATABASE_PASSWORD ?? '',
  database: process.env.DATABASE_NAME ?? 'v84',
  entities: [__dirname + '/../**/entities/*.entity{.ts,.js}'],
  migrations: [__dirname + '/migrations/*{.ts,.js}'],
  synchronize: false,
  timezone: 'Z',
});
```

## Database folder structure

Everything lives under `src/database/` inside the API app:

```
apps/api/src/database/
├── database.module.ts       ← NestJS module with TypeORM config
├── data-source.ts           ← CLI-only data source (migrations, seeds)
├── seed-runner.ts            ← discovers and runs seeders idempotently
├── migrations/               ← GENERATED by typeorm migration:generate
│   └── 1775507275439-InitialSchema.ts
├── seeds/
│   └── demo.seeder.ts       ← seeder class with run(dataSource)
└── factories/
    ├── spin.factory.ts       ← createSpin(ds, overrides)
    └── index.ts              ← barrel export
```

## Seed runner — idempotent via seed_history table

Tracks which seeders have run in a `seed_history` table. Re-running is safe — already-executed seeders are skipped.

```typescript
// database/seed-runner.ts
import 'dotenv/config';
import { readdirSync } from 'fs';
import { join, basename } from 'path';
import { DataSource } from 'typeorm';
import { AppDataSource } from './data-source';

interface Seeder {
  run(dataSource: DataSource): Promise<void>;
}

async function ensureSeedHistoryTable(ds: DataSource): Promise<void> {
  await ds.query(`
    CREATE TABLE IF NOT EXISTS seed_history (
      name VARCHAR(255) NOT NULL PRIMARY KEY,
      executed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
    )
  `);
}

async function hasRun(ds: DataSource, name: string): Promise<boolean> {
  const rows = await ds.query('SELECT 1 FROM seed_history WHERE name = ? LIMIT 1', [name]);
  return rows.length > 0;
}

async function markAsRun(ds: DataSource, name: string): Promise<void> {
  await ds.query('INSERT INTO seed_history (name) VALUES (?)', [name]);
}

async function main() {
  await AppDataSource.initialize();
  await ensureSeedHistoryTable(AppDataSource);

  const seedsDir = join(__dirname, 'seeds');
  const files = readdirSync(seedsDir)
    .filter((f) => f.endsWith('.seeder.ts') || f.endsWith('.seeder.js'))
    .sort();

  for (const file of files) {
    const name = basename(file).replace(/\.seeder\.(ts|js)$/, '');
    if (await hasRun(AppDataSource, name)) {
      console.log(`skip: ${name}`);
      continue;
    }
    const mod = await import(join(seedsDir, file));
    const SeederClass = Object.values(mod).find(
      (v): v is new () => Seeder => typeof v === 'function' && v.prototype?.run,
    );
    if (SeederClass) {
      console.log(`run: ${name}`);
      await new SeederClass().run(AppDataSource);
      await markAsRun(AppDataSource, name);
    }
  }

  await AppDataSource.destroy();
}

main().catch((err) => { console.error(err); process.exit(1); });
```

## Factories — one per entity

Factory functions create test data. Accept `overrides` for customization, use sensible defaults.

```typescript
// database/factories/spin.factory.ts
import { v4 } from 'uuid';
import { DataSource } from 'typeorm';
import { Spin } from '../../apps/api/src/entities/spin.entity';

export function createSpin(ds: DataSource, overrides: Partial<Spin> = {}): Promise<Spin> {
  const repo = ds.getRepository(Spin);
  return repo.save(
    repo.create({
      key: v4(),
      number: Math.floor(Math.random() * 10000),
      ...overrides,
    }),
  );
}
```

```bash
# Run seeds
pnpm ts-node database/seed-runner.ts
```

## Seed user accounts — predictable dev credentials

When seeding user/account entities, create one account per role with predictable credentials. Format:
- Email: `{role}@{role}.localhost`
- Password: `{role}`

```typescript
// In demo seseeder — create one account per role
const roles = [
  { email: 'admin@admin.localhost', password: 'admin', role: 'admin' },
  { email: 'user@user.localhost', password: 'user', role: 'user' },
  { email: 'moderator@moderator.localhost', password: 'moderator', role: 'moderator' },
];

for (const { email, password, role } of roles) {
  await createAccount(ds, { email, passwordHash: await hash(password, 10), role });
}
```

This applies to any project with auth. Adjust roles to match the project (admin/user, owner/member, etc). Every role in the system gets a seed account — no exceptions. Seeder should log credentials so devs can find them easily.
