# Agent Architecture

> How agents are loaded, what they read, how they communicate.

## Token-efficient loading

Every agent call is short and flat. The system prompt is the agent's `agent.md` (or inline variant), the user prompt is the context the scripts pre-bundle. There are **no** tool calls inside a pipeline agent — only the executor uses tools.

```
system message:  agent.md (identity + skill list, ~200 tokens)
user message:    composed by scripts/*/call.sh:
                   - skill instructions (shared/skills/*.md or architect/skills/*.md)
                   - context/{role}/{topic}/identity.md        ← role + topic identity
                   - context/{role}/packages.md                ← installed deps
                   - relevant trees, conventions, plan text, current draft, corrections
agent returns:   plain text
scripts write:   plan/{n}/raw/{role}:{topic}.md                ← full response
                 plan/{n}/{role}/{topic}.md                    ← parsed entries
```

Context is regenerated by `scripts/build-context.sh` and by every script that changes plan state, so it's always current.

## Context files

```
v84-docs/context/
├── architect/
│   └── packages.md                 ← root-level package summary
├── executor/
│   └── packages.md                 ← flat package summary across all workspaces
├── back-nestjs/
│   ├── packages.md                 ← apps/api/package.json summary
│   ├── topics.md                   ← active topics overview
│   ├── api/
│   │   ├── identity.md             ← composed role+topic identity
│   │   └── corrections.md          ← created during patch rounds
│   ├── entities/
│   ├── services/
│   └── …
├── front-nextjs/…
├── ops/…
└── reviewer/…
```

Each `{role}/{topic}/identity.md` bundles:

- Shared conventions + role conventions extracted from `structure/`
- Topic scope (what this agent owns, what it does NOT own)
- Tree filtered to tagged files that matter for the role
- The current iteration plan text
- The current draft (only in patch rounds)
- Architect corrections for this topic (only in patch rounds)

Leads get a role-scoped version that points at all drafts for their role. The architect gets everything — all plan drafts + all lead reviews. When a role's lead reports no corrections in a round, the architect receives an explicit `CLEAN — this lead reported no corrections this round.` marker for that role rather than an empty section, so "all leads clean" is never indistinguishable from "leads didn't run".

## Agent files

Every `agent.md` is minimal (~200 tokens): identity, tag, skill list. No convention references, no authoritative sources. Everything comes from context.

Two flavours:

- `agent.md` — canonical form used when the agent lives on its own (e.g. via Claude Code skills).
- `agent-inline.md` — a flatter variant for call paths that inline everything into one system prompt (the bash pipeline does this).

Shared skills live in [v84-docs/agents/shared/skills/](../agents/shared/skills/):

- `draft-inline.md` — topic agent writes a draft for its topic.
- `lead-review-inline.md` — lead reviews all of its role's drafts.
- `patch-inline.md` — agent applies corrections to its own draft.

Architect skills live in [v84-docs/agents/architect/skills/](../agents/architect/skills/). Both are inline variants called exclusively by scripts — the user never invokes the architect:

- `plan-inline.md` — decompose user request into `plan/{n}.md`.
- `review-inline.md` — validate leads, cross-role check.

Executor skills live in [v84-docs/agents/executor/skills/](../agents/executor/skills/). The executor is the only agent a user talks to:

- `init.md` — first-run guided setup for a brand-new project (roles, topics, scaffold).
- `run.md` — pipeline driver. State check + dispatch the scripts under `v84-docs/scripts/` (plan → cycle → extract → promote). The interactive entry point once initialised.
- `execute.md` — the only skill that writes source code. Uses tools. Invoked from `run.md` once `plan/{n}/tasks.md` exists.

## Convention files

All project rules live under `structure/conventions/`:

- `structure/conventions.md` — shared rules (naming, IDs, brand, testing philosophy).
- `structure/conventions/back-nestjs.md` — backend-specific.
- `structure/conventions/front-nextjs.md` — frontend-specific.
- `structure/conventions/ops.md` — ops-specific.
- `structure/conventions/reviewer.md` — reviewer-specific.
- `structure/conventions/executor.md` — code patterns + tagging rules used during Step 8.

**Maintenance rule:** when the stack changes, update the relevant convention file. The next `build-context.sh` run propagates the change into every agent's context.

## Key rules

- **One context per agent.** Scripts bundle everything; the agent reads once.
- **One agent, one topic, one call.** Never batch topics into a single agent — output quality drops sharply when topics cross.
- **No tool calls in pipeline agents.** The scripts handle all IO. Only the executor agent (Step 8) uses tools, because it writes actual source code.
- **System prompt stays small.** Keep `agent.md` under ~200 tokens — all of the "what to do" lives in the skill prompt and the context.
- **Context is always current.** After every script that mutates plan state, `build-context.sh` re-runs. Drafts, corrections, verdicts — whatever exists at that moment is in context next call.
- **Topic scoping is strict.** A topic agent must ignore work that belongs to another topic, even if it notices a problem. Leads raise cross-topic issues within a role; the architect raises cross-role issues.
